Swift 异常断点 （Swift Error Breakpoint）

抛出一个继承自 ErrorType 类型的错误时会触发这个断点

异常断点（Exception breakpoint)

经常有崩溃直接跳到了 main 函数里，这个时候加一个全局的异常断点，可以快速定位到错误信息。

  • 推荐将 异常断点 加到 User 中，这样每个项目中都有了异常断点。
  • 与此同时也建议加一个 Debugger Command 用 bt 打印出方法调用的堆栈信息。

符号断点（Symbolic breakpoint)

可以监听方法、函数的执行。比如可以监听 viewDidLoad， 每次页面加载就会进入断点。这对第一次看一个项目或者 Demo 很有帮助。objc_exception_throw 监听这个方法可以在系统捕获异常时，进入断点。

其它

在控制台 Watch 某个对象，当 num 的值发生变化的时候，会进入断点并在控制台打印出来。

LLDB

大家平时用 LLDB 一般就是 p 和 po 吧，其实最实用的也就这两个。
具体一些直接使用 help

print

  • print 打印需要查看的变量，例如 print num
  • LLDB 会做前缀匹配，print 还能使用简写prin, pri, p
  • 不能使用 pr，因为 LLDB 不能消除和 process 的歧义
  • po(print object)可以打印对象的 description 方法的结果
  • 打印不同格式可以用p/x number打印十六进制，p/t number打印二进制，p/c char打印字符。这里是完整清单
  • po [[[UIApplication sharedApplication] keyWindow] recursiveDescription] 打印出当前的视图层级

expression

  • expression 可以改变一个值，例如expression num 实际上它改变了程序中的值
  • e -p — dataArray 也可以打印对象的description方法的结果，等同于po

bt
打印堆栈信息

断点
continue /c 程序继续运行
next / n 单步运行
step / s 进入某个方法
finish 返回到上层方法

Chisel

Chisel 其实就是一个 Python 脚本的集合，这些脚本拼接 (命令) 字符串，然后让 LLDB 执行，所以如果你会使用 Python 的话，那么你可以根据自己的使用习惯封装一些常用的 LLDB 命令。

Installation

在根目录下创建.lldbinit,复制 command script import /usr/local/opt/chisel/libexec/fblldb.py 进 .lldbinit 文件中，当然还需要重启一些 Xcode。

pviews / pvc
查看 view / viewController 的层级 类似的命令还有 ptv / pcells

pclass

fv / fvc
通过类名搜索当前内存中 view / viewController 的对象（可以配合正则表达式使用）

visualize

border / unborder

hide / show

taplog

internals
打印一个对象内部的成员变量

bmessage
这个命令就是 LLDB 添加一个断点，譬如-viewWillAppear:这个方法，在当前控制器中你没有实现它，但是你又想在调用它的时机触发中断。

exception bad access
虽然现在开启了 ARC，很难再遇到 EXC_BAD_ACCESS 错误了，但是在某些确定的情况下，该错误还是会发生的，这时候开启 Zombie 可以快速找到被释放的对象。原理是：当我们release内存中的一个对象时，Zombies 会将对象标记为release，实际上该对象还停留在内存中。当我们再次访问这个被 release 的对象时，Xcode 可以告诉我们正在访问的对象是一个不应该存在的对象了。Notification与多线程
